# Expert Agent 面板 PRD（v1.0）

## 1. 背景与目标
当前团队通过飞书表格管理多项目标注产量、准确率、POC 评分，流程分散在脚本与手工操作中，存在：
- 入库标准不统一，列名变化导致识别不稳定
- 项目/子项目人员统计不一致，返工频繁
- 面板访问链路不稳定（临时隧道失效）

本产品目标是提供一个统一的「作业数据入库 + 质量统计 + 项目看板」平台，提升稳定性与效率。

## 2. 产品目标（SMART）
- 入库成功率：>= 95%
- 列识别正确率（初标/质检/结果）：>= 90%
- 面板可用性：工作时段 >= 99%（内网或固定部署）
- 单项目处理时长：< 3 分钟（不含外部 API 抖动）

## 3. 用户与角色
- 运营/项目经理：创建项目、执行入库、查看项目详情
- 质检/POC：查看产量、准确率、POC 评分明细
- 工程支持：维护 token、部署、排障

## 4. 核心场景
1. 用户输入项目名称 + 飞书链接 + SOP + POC，执行入库  
2. 系统自动识别表头与角色列，计算产量/准确率并写入数据库  
3. 用户在面板查看项目总览、子项目详情、人员详情  
4. 低置信识别或无质检场景下，按规则回退并给出可解释日志  

## 5. 功能范围

### 5.1 已有核心能力（需稳定）
- 飞书读取：支持 `sheets`、`wiki?sheet=...`
- 统计计算：初标、质检、POC、项目汇总
- 面板展示：项目列表、详情、基础过滤
- SQLite 入库：`projects/runs/person_metrics_base/project_metrics_base`

### 5.2 本期必须完成
- 通用列识别（Ark 主导）：
  - 支持多行表头（group::leaf）
  - 关键词覆盖 `gt/qc/cc/qa/review/质检/复核/抽检`
  - 仅客观列参与 pass/fail
- 原结果列优先策略：
  - 原结果列可用时不覆盖
  - 低置信时回退原结果列；无原结果列则不计准确率
- 人员列安全约束：
  - 初标/质检必须通过人名校验（中文姓名/拼音/英文名）
  - 禁止将标签类字段写入人员统计
- token 自动刷新：
  - 支持 `user_token_file` 读取与刷新落盘
  - 过期自动刷新，失败明确提示

### 5.3 本期不做
- 多租户权限体系
- 数据库迁移到 MySQL/PG
- 完整工作流编排（仅保留脚本 + 面板）

## 6. 关键流程（E2E）
1. 用户在面板输入：项目名、源表 URL、SOP、POC、鉴权信息  
2. 系统解析 URL -> 获取 sheet 数据 -> 自动识别表头深度  
3. Ark 识别 source/reference 列组 -> 行级判定 comparable/pass  
4. 生成初标/质检/POC 与项目汇总指标  
5. 写入 SQLite，刷新面板视图  
6. 若失败：记录错误原因与可执行修复建议

## 7. 数据口径
- 产量：角色列非空行数
- 被质检数：初标行中可判定质检结果的行数
- 质检通过数：`judge_pass_label == True` 的行数
- 准确率：通过数 / 被质检数（分母为 0 时显示“无数据”）
- 主观列（如 description/comment）不参与准确率

## 8. 非功能需求
- 性能：10k 行表格处理 < 60s（网络正常）
- 可观测性：关键阶段日志（读取、识别、判定、入库）
- 稳定性：网络失败重试、超时兜底
- 安全性：禁止暴露未授权公网入口；token 不落日志明文

## 9. 风险与对策
- 风险：飞书表头高度异构 -> 对策：Ark + 关键词 + 低置信回退
- 风险：token 过期/失效 -> 对策：自动刷新 + 明确错误码
- 风险：临时隧道不稳定 -> 对策：优先固定部署（EdgeOne/内网）
- 风险：人员识别误判 -> 对策：人名校验 + 规则拦截

## 10. 验收标准（UAT）
- 能成功入库至少 10 个不同结构项目
- `DL-052` 这类“无质检”项目不会误产生质检准确率
- 人员详情不再出现 `CV - xxx` 等标签名称
- token 过期后可自动刷新并继续执行
- 面板支持稳定查看项目与子项目详情

## 11. 里程碑
- M1（D+2）：列识别/结果回退/人名校验稳定
- M2（D+4）：token 自动刷新与异常处理闭环
- M3（D+5）：UAT 回归 + 发布说明 + 运维手册
